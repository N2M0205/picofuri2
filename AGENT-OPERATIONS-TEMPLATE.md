# Claude Code エージェント化 運用テンプレート

ピコフリ2プロジェクトで実運用・検証済みの設計。他プロジェクトへの
導入時は {PROJECT} を実際の名前に置き換えて使用する。

## 1. 基本構造：三者分業モデル

| 役割 | 担当 | 責務 |
|---|---|---|
| 設計担当 | Claude（チャット） | 方針決定・指示文作成・成果物レビュー・Docs確認 |
| 実行担当 | Claude Code（実行環境常駐） | 実装・テスト・デプロイ・調査・報告 |
| オーナー | 人間 | 指示文の中継、ビジネス判断、最終承認 |

核心は「実行担当に判断させる範囲」の明文化。丸投げでも完全管理でもなく、
判断の種類ごとに自走可否を分ける。

## 2. 判断領域分類ルール（自走の境界線）

CLAUDE.mdに明記し、実行担当が自動遵守する。

| 領域 | 例 | 実行担当の動き |
|---|---|---|
| ビジネス判断 | 価格閾値、対象基準、通知方針、削除・統合の可否 | 停止してオーナーに確認 |
| 技術選択 | 変数名、実装方式、ライブラリ選定、テスト手法 | 仮定して進行し、前提ログに記録 |
| 迷う場合 | — | ビジネス判断側に倒して停止 |

## 3. 暴走防止の技術的ガードレール

- Gitフック強制: mainへの直接コミット禁止、pushは OWNER_APPROVED=1 必須
  （口約束でなく技術的にブロック）
- タスク区切り: 3タスクごとに停止・報告、調査15分超で中間報告、
  エラー3連続で停止
- 事前承認済み定型手順: 頻出する安全な手順は「事前承認済み」として
  都度確認を不要化。緊急停止手順も事前承認で即応可能に
- バッチ限定ルールによる上書き: 複数タスクの連続自走を許可する場合、
  オーナー承認付きで「3タスク停止」をマイルストーン報告に代替できる。
  ただし影響の大きい操作（本番DB書き込み等）の直前には必ず
  dry-run提示→承認待ちの停止ポイントを1箇所埋め込むこと

## 4. 指示文のライフサイクル管理（3状態）

設計担当が作成した指示文は、必ずしも実行担当に届くとは限らない。
オーナーの中継には以下の3状態があり、いずれも正当な選択である。

| 状態 | 意味 | 扱い |
|---|---|---|
| ✅送信済み | 実行担当に貼り付けた | 通常フロー |
| ⏸保留 | 意図的に送っていない（脱線アイデア・時期尚早・優先度変更等） | BACKLOGの「⏸保留中指示」に理由付きで記録 |
| 🗑破棄 | 検討の結果不要と判断 | 記録不要（割り切ってよい） |

運用ルール:
- 設計担当は指示文に短いタイトルを付ける（後日の照合を容易にする）
- 設計担当は節目（マイルストーン完了時等）に未確認の指示文の状態を
  オーナーに確認する。「保留」の回答を受けたら以後の再確認はしない
- チャットの脱線から生まれた指示文は保留・破棄が多くて正常。
  全てを送信する義務はない
- 未送信の検知は設計担当（チャット側）の責務。実行担当は
  「一度も送られていない指示」を原理的に知り得ないため、
  実行担当側の棚卸し（後述）はあくまで補完である

## 5. 報告の3層構造

| 報告の種類 | 経路 |
|---|---|
| 長文の実装結果・調査レポート | Google Docsに書き込み、Telegramにリンク+要約 |
| 短い確認・分岐選択（Q1/Q2形式） | ターミナル/チャットで直接 |
| 完了・入力待ちの状態通知 | Stop/Notificationフック → Telegram自動通知 |

設計担当はDocsを直接読んで確認する（オーナーのコピペ中継を最小化）。

## 6. 記録の5ファイル体制

| ファイル | 役割 |
|---|---|
| CLAUDE.md | 恒久ルール本体（実行担当が自動で読む） |
| {PROJECT}-BUSINESS-RULES.md | 確定したビジネス判断のみ |
| {PROJECT}-BACKLOG.md | 未着手タスク + ⏸保留中指示 |
| logs/assumptions/ASSUMPTIONS-{YYYY-MM}.md | 技術選択の仮定記録（月次） |
| セッションログ / Docs | 作業の詳細記録 |

## 7. 完了報告の必須フォーマット

- 完了した作業・変更ファイル・コミットハッシュ
- テスト結果
- 前提ログ（仮定して進めた項目。ゼロでも「なし」と明記）
- 次のアクション or 停止ポイントの明示

## 8. セッション開始時の棚卸し（実行担当）

新規セッション or 長い間隔後に5-10分で実施:
1. git log --oneline main -20 で直近コミット確認・報告
2. git branch -a で仕掛かり中ブランチ列挙・報告
3. 直近のBACKLOG/ASSUMPTIONS-LOG追記を確認・報告
4. 「送られたはずだが未着手に見える指示」があれば明示し確認を仰ぐ
   （※一度も送られていない指示は検知不能。それは設計担当の台帳管理で補う）
5. node scripts/check-yahoo-breaker.js で直近24時間のYahoo breaker
   発動有無を確認・報告（2026-07-08 picofuri2プロジェクトで追加、
   類似の見落とし事故対策として汎用化）

## 9. 運用して分かった教訓

- 「指示文の提示」≠「送信済み」: 送信漏れ事故が複数回発生。
  対策は §4 のライフサイクル管理と §8 の棚卸しの二重化。
  それでも防ぎきれない場合がある（picofuri2プロジェクトで5回発生）ため、
  重要な指示文は「提示後、次のメッセージで送信有無を確認する」運用を徹底する
- 単一コマンドによる状態確認の危うさ: grepの複数ファイル指定でfilenameが
  prefixされ、集計ロジックが誤動作した事例あり。重要な安全装置の状態確認は
  専用スクリプト化し、単発コマンドに頼らない
- 実行担当は「オーナーの仮説」を実データで覆してよい: 想定と実測が
  食い違ったら実データ優先で逆の提案を歓迎する
- 自走バッチには意図的な停止ポイントを1つ埋め込む（§3参照）

## 10. 最小構成での導入手順

1. CLAUDE.mdに §2 判断領域分類 + §3 タスク区切りを書く
2. Gitフックを入れる
3. §7 完了報告フォーマットを定義する
4. 慣れてきたら §5 Docs連携・Telegram通知フックを追加

1-3は初日に導入可能。4以降は運用が回り始めてから。
