# 開発環境 安全性テンプレート

※ 判断層 (誰が何を決めるか) は `AGENT-OPERATIONS-TEMPLATE.md` を参照。
本ファイルは技術層 (どう安全に実行環境を作るか) を扱う。両者は補完関係。

## 0. 最初に確認すること：本当にDocker隔離が必要か

新規プロジェクトのVPS環境構築時、まず以下を確認する：

```bash
whoami
sudo -n -l   # パスワードなしsudoの範囲確認
ls -la /home/{user}  # 書き込み可能範囲の確認
```

**判断基準**:
- 一般ユーザー（root以外）で運用し、かつpasswordless sudoが設定されていなければ、
  Claude Codeの`--dangerously-skip-permissions`はOS権限昇格を意味しない
  （Claude Code内部の確認プロンプトスキップに過ぎない）
- この場合、書き込み可能範囲は自ユーザーのホームディレクトリ + /tmp に限定され、
  Docker隔離を導入しなくても実用上安全な水準に収まる（picofuri2プロジェクトの
  実例で検証済み、2026-07-09調査）
- 逆に、常にrootでログインし作業している場合は、①即座に非rootユーザーへの
  切り替えを検討する、②切り替えが困難な事情がある場合のみ Docker隔離
  （下記1）を導入する、の優先順で対処する

## 1. 【防壁・実験室】Docker（隔離コンテナ）－ root運用が避けられない場合のみ

* 目的: rootでの作業が構造上避けられない場合に、Claudeの暴走からVPS本体を守る
* 実装: VPS内にDockerを導入し、Claude Codeや各種言語環境をコンテナ内で動かす
* 注意: コンテナに本尊のディレクトリをマウントしている場合、そこは保護されない。
  「1ミリも傷つかない」は誇張であり、マウント範囲の設計が安全性を左右する

## 2. 【命綱・タイムマシン】Git / GitHub（バージョン管理）

* 目的: 「戻せない恐怖」をゼロにし、AIへの指示をタスク単位で切り分ける
* 実装（picofuri2で実績あり）:
  * main への直接コミット・pushを禁止するGitフック（pre-commit, pre-push）
  * push時に `OWNER_APPROVED=1` のような明示的承認フラグを必須化
  * ブランチ命名規則の統一（feat/, fix/, docs/等）
  * マージ前に必ず `git diff main` で差分提示

## 3. 【自動化ロボ】CI/CD（自動テスト・自動デプロイ）

* 目的: テスト漏れをなくしつつ、人間の最終承認ゲートは残す
* 重要な調整: 完全自動デプロイは、Gitの承認ゲート（2の仕組み）と衝突する。
  推奨構成は「CI（自動テスト）は全面導入し、CDの直前に人間承認を挟む」
  * GitHub Actions等でpush時に自動テストを実行
  * テスト通過後、デプロイは「実行可能」状態になるが、実際の反映は
    承認コマンド（例: OWNER_APPROVED=1 での手動デプロイトリガー）を要する

## 4. 【設計図の自動化】IaC（構成のコード化）

* 目的: 環境構築手順を再現可能にする
* 実装: docker-compose.yml や Ansible playbook で環境構成を記述
* 優先度: 「今すぐ壊れる想定」がない限り、VPS再構築時の保険として
  後回しでよい（0のroot/非root判断、2のGit運用の方が優先度が高い）

## 5. 判断フローまとめ
新規プロジェクト開始
↓
Q: 常にrootで作業する必要があるか？
NO → 非rootユーザー運用 + Git運用（2）のみで実用上安全
Docker（1）は不要、CI/CD（3）・IaC（4）は任意で拡張
YES → Docker隔離（1）を優先実装、その上でGit運用（2）を重ねる

## 6. 参考実例：picofuri2の非root運用検証（2026-07-09）

* 環境: picofuri2ユーザー（一般ユーザー）でClaude Code運用、
  `--dangerously-skip-permissions`使用
* 検証結果: passwordless sudo なし、書込可能範囲は
  `/home/picofuri2` + `/tmp` のみ、本セッション中のsudo成功実績0件
* 結論: Docker隔離なしでも実用上安全な構成として機能した
